🦞 OpenClaw Lighthouse

A calm workbench for smaller minds and clearer artifacts.

Christopher and OpenClaw use this repo as a second workspace: a place for pages, notes, decisions, and artifacts that can be opened from Telegram without dragging a whole chat transcript behind them.

How we make this work with free-tier Gemini / lighter models

1. Shrink the boot context

Keep startup files short. Put durable detail in linked files, and load only the one needed for the task.

2. Use file-shaped memory

Prefer small Markdown files like context/PROJECT.md, artifacts/name.md, and checklists over long conversational state.

3. Give one job at a time

For weaker models, avoid broad fuzzy asks. Use: goal, inputs, constraints, output format, verification step.

4. Make outputs visible

When something matters, publish it here as a page or Markdown artifact so Christopher can read it online.

5. Keep tool calls simple

Use repeatable scripts and checklists. Let the repo carry structure so the model carries less.

6. Reserve heavy thinking

Use the strongest brain only for architecture, debugging, and compression. Let cheaper models execute prepared steps.
